# Photo Album Viewer 项目规则

## 版本号维护

- 每次功能、修复或配置改动交付前，必须根据改动大小更新本项目版本号，并同步 `package-lock.json` 与 `CHANGELOG.md`。
- 版本号遵循语义化版本：
  - `patch`：向后兼容的 bug fix。
  - `minor`：向后兼容的新功能。
  - `major`：破坏兼容性的变更。
- 版本号修改应与对应功能或修复一起纳入交付；若用户要求先拆分提交，则按用户要求拆分。

## 构建验证

- 日常开发交付前，必须运行与改动范围匹配的测试；涉及前端打包链路、模块导入、webpack 配置或发布前检查时，至少运行 `npm run build:webpack`。
- 需要生成可安装应用、验证 Electron 打包、修改 `package.json` 的 `build` 配置、调整主进程/预加载打包相关代码，或用户明确要求“打包”“构建安装包”“npm run build”时，必须运行 `npm run build`。
- 已合并到 `main`、且用户需要通过本地安装包才能实际使用的可视功能或修复，交付前必须运行 `npm run build`，不能只运行 `npm run build:webpack`。
- `npm run build` 生成的 `dist/` 产物默认视为本地构建输出，不纳入提交，除非用户明确要求保存或发布这些产物。
- 如果 `npm run build` 因签名证书缺失只跳过 macOS 代码签名但产物成功生成，应记录为分发风险，不视为构建失败。

## 分支管理

本项目是个人项目，默认由 Agent 负责日常分支操作。目标是让 `main` 始终代表已完成、已验证、可继续开发的稳定基线，同时让每个功能或修复都有清晰的独立分支边界。

### 分支角色

- `main`：稳定主线，只合入已经完成、通过验证、版本号和 `CHANGELOG.md` 已同步的改动。
- `codex/<task-slug>`：Agent 开发分支。每个明确功能、修复或配置调整默认使用一个独立分支。
- 除非用户明确要求，不创建长期并行开发分支、release 分支或实验分支。

### 开始新任务

- 开始代码、配置或文档改动前，先检查当前分支和工作区状态。
- 如果当前在 `main` 且工作区干净，应从 `main` 创建新的 `codex/<task-slug>` 分支。
- 如果当前在 `codex/<task-slug>` 且该分支尚未完成，应继续在当前分支工作。
- 如果当前在 `codex/<task-slug>` 但用户确认上一项任务已经完成，且工作区干净、测试通过，应先合并回 `main`，再从 `main` 创建新分支。
- 如果工作区存在未提交或来源不明的改动，Agent 不得直接合并、切分或覆盖；必须先说明状态，并只处理可明确归属的本轮改动。

### 分支命名

- 默认格式：`codex/<scope>-<short-action>`。
- 名称使用小写英文、数字和连字符，例如 `codex/dual-page-viewer`、`codex/delete-image-flow`。
- 分支名应描述用户可感知的任务，不使用 `temp`、`test`、`fix2` 这类含糊名称。

### 提交与合并

- 一个分支可以包含多个小提交，但最终必须能作为一个完整任务理解。
- 每次交付前必须运行与改动范围匹配的验证；无法验证时要明确说明原因和风险。
- 合并回 `main` 前必须确认：
  - 工作区干净，或仅有明确属于本任务且已提交的改动。
  - 测试或等价验证已经通过。
  - 版本号、`package-lock.json` 和 `CHANGELOG.md` 已按项目规则同步。
- 个人项目默认使用本地 `git merge --no-ff <branch>` 合并到 `main`，保留任务分支边界。
- 合并完成后，下一项任务必须从更新后的 `main` 新建分支，不在已完成分支上继续叠新需求。

### 远端同步

- 默认不自动 `push`、不自动创建 Pull Request，除非用户明确要求。
- 如果本地 `main` 与 `origin/main` 不一致，Agent 在 push 或 PR 前必须先说明差异和建议路径。
- 用户要求发布、备份或跨设备同步时，再按当前分支状态执行 push。

### 分支收尾

- 已合并的 `codex/<task-slug>` 分支可以保留作为历史记录；只有用户明确要求清理时才删除。
- 如果合并后发现问题，应从最新 `main` 新建修复分支，不直接修改已经收尾的旧分支。
